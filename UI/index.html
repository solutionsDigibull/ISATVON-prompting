How it works

The structure covers the answer as well as the ask

Most frameworks structure your prompt and hope for the best. ISATVON also contracts the response, so a misread task, a broken constraint, or a silent assumption is visible in one line.

The framework

Seven sections. Both directions.

Every section works twice: once in your prompt, once in the model's response.

I

Instructions

In your prompt
Role, task, rules.
In the response
The task as it understood it.
S

Source

In your prompt
Context to use, gaps not to fill.
In the response
Sources it actually used.
A

Automation

In your prompt
Method + self-verification step.
In the response
How it verified the result.
T

Tech stack

In your prompt
Capabilities allowed / forbidden.
In the response
Tool policy honored, no fabrication.
V

Variables

In your prompt
Measurable constraints + fallback.
In the response
Constraints honored or broken, with reason.
O

Outcome

In your prompt
Exact response contract.
In the response
The deliverable itself.
N

Notification

In your prompt
Required meta-reporting.
In the response
Assumptions, confidence, omissions.

Before / After

"Keep it short" is not a constraint

One real conversion from the examples library.

Raw prompt
"Summarize this report for me, keep it short."
ISATVON
After conversion
  • "Short" becomes max 150 words with a declared fallback instead of a silent guess.
  • The model must verify every sentence traces to the report before answering.
  • The reply opens with the task as understood, so a misread is visible in one line.

When not to use it

✕
One-shot stylistic asks A tweet rewrite or a tone change doesn't need seven sections; plain COSTAR (or nothing) is fine.
✕
Conversation This structures deliverables, not chat.
✕
Agent loop contracts If you need a bounded, machine-validated retry loop (JSON manifest, MCP tools, guaranteed exits), that's the ISATVON specification and its validator, not this.
